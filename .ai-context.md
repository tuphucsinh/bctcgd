# Project Context: Báo cáo Tài chính Gia đình (BCTCGD)

## 1. Vision & Purpose (Mục tiêu)
- **Problem:** Quản lý thu chi, nợ, tiền lãi và phân bổ tài sản của cấu trúc gia đình bị phân mảnh, khó nhìn tổng quan.
- **User:** Gia đình (Tài khoản riêng biệt: Hiếu, Ly. View "Gia đình" (GĐ) gộp dữ liệu cả hai).
- **Core Value:** Giao diện Dashboard Bento Grid hiện đại (Dark Mode "Premium"), tích hợp nhập liệu tức thời, báo cáo trực quan cho phép chuyển user 1 click (Hiếu/Ly/GĐ).
- **Các tính năng khác:** Theo dõi dư nợ gốc, chi phí lãi vay, phân bổ tài sản (BĐS, Cổ phiếu, Tiền mặt).

## 🛠️ Cấu trúc Database (v2) - CHI TIẾT
### 1. Phân loại Thu nhập & Chi phí (Categories)
- **Thu nhập (Income)**: 
  - *Chủ động*: Lương, Thưởng.
  - *Thụ động*: BĐS, Crypto, Vàng, Tài chính, Khác (Cần tính toán cho Gauge "Tự do tài chính").
- **Chi tiêu (Expense) - 10 mục**: 
  - Ăn uống, Di chuyển, Hóa đơn, Giao tế, Giải trí, Đồ dùng gia đình, Y tế, Học hành, Con cái, Khác.

### 2. Các loại Tài sản (Assets)
- **5 Nhóm chính**: Bất động sản (REAL_ESTATE), Tài chính (FINANCE), Crypto (CRYPTO), Vàng (GOLD), Khác (OTHER).
- **Trường dữ liệu**: Tên, Nhóm, Phân loại (LIQUID/FIXED), Số lượng, Giá mua, Đơn giá hiện tại, Tổng giá trị, Ghi chú, Ngân hàng (nếu có).
- **Logic**: 
  - Hỗ trợ **Bán một phần (Partial Asset Sale)**: Cho phép trừ số lượng lẻ, tự động tính lại `current_value` và giữ trạng thái `ACTIVE`. Nếu bán hết sẽ chuyển sang `SOLD`. Khi bán thành công sẽ tự động chèn 1 bản ghi vào `transactions` (Lãi/Lỗ) và **cộng tiền mặt** tương ứng giá trị bán.
  - Checkbox "Trừ tiền mặt" (mặc định) khi thêm tài sản -> Tự động tạo giao dịch `ASSET_BUY` và trừ số dư Tiền mặt hiện có.
  - **Manual Cash Update**: Nút "+" tại thẻ Tiền mặt trên Dashboard cho phép điều chỉnh số dư tay mà không tạo giao dịch (trong trường hợp sai lệch nhỏ hoặc khởi tạo).

### 3. Quản lý Nợ & Cho vay (Debts)
- **Nợ (Borrow)**:
  - *Danh mục*: Nợ Ngân hàng, Cọc BĐS, Nợ khác.
  - *Phân loại*: Nợ ngắn hạn, Nợ dài hạn.
  - *Lãi suất*: Hỗ trợ hiển thị lãi suất (%/năm) ngay dưới tên khoản nợ.
  - *Database Schema*: `id`, `name`, `debt_type` ('BORROW'/'LEND'), `initial_principal` (Số gốc ban đầu), `remaining_principal` (Số nợ còn lại), `interest_rate`, `monthly_payment_amount`, `term`, `created_at`.
- **Cho vay (Lend)**:
  - *Dữ liệu*: Tên, Số tiền (`remaining_principal`), Lãi suất, Ghi chú.
- **Logic Owner**: Khoản Nợ (Debts) và Tài sản (Assets) được tính là **CỦA CHUNG GIA ĐÌNH**, không có thuộc tính owner (loại bỏ phân chia HIEU/LY để đơn giản hóa việc quản lý dòng tiền và tài sản).
- **UI Layout**: Hiển thị dạng 2 cột riêng biệt (Nợ bên trái, Cho vay bên phải) trên desktop để tối ưu so sánh.

### 4. Giao dịch (Transactions)
- **Loại**: INCOME, EXPENSE, DEBT_PAYMENT, ASSET_BUY, ASSET_SELL, **ADJUSTMENT**.
- **Dữ liệu**: Số tiền, Ngày, Người chi (HIEU/LY/JOINT), Danh mục, Ghi chú.
- **Logic Owner**: Chỉ CÓ Giao dịch (Transactions) mới phân chia `HIEU`/`LY` (để theo dõi ai chi tiền, ai thu tiền). Nợ và Tài sản sẽ KHÔNG CÓ owner. Màn hình "Gia đình" sẽ gộp toàn bộ giao dịch, còn Tài sản và Nợ thì luôn hiển thị đầy đủ cho bất kỳ ai.
- **ADJUSTMENT Type**: Dùng cho các thao tác điều chỉnh số dư tiền mặt thủ công hoặc khởi tạo nợ dính dáng đến tiền mặt. Note mặc định: `"điều chỉnh tiền mặt"`.
- **Liên kết**: `is_principal_payment` (để phân biệt trả gốc và lãi nợ).

## 2. Tech Stack (Công nghệ)
- **Frontend:** Next.js 15 (App Router - sử dụng Async `searchParams`), React 19, Tailwind CSS v4.
- **State Management:** React Hooks.
- **Database:** Supabase (PostgreSQL). Thao tác thông qua `@supabase/ssr` và `@supabase/supabase-js` (cho Admin context).
- **Next.js 15 Notes**: `searchParams` và `params` trong Page/Layout là **Promise**. Phải `await` trước khi sử dụng.
- **Unique Libs**: 
  - `framer-motion`: Hỗ trợ Animation cao cấp.
  - `sonner`: Component hiển thị thông báo toast.
  - `recharts`: Cung cấp biểu đồ trực quan.
  - `lucide-react`: Bộ icon giao diện.
  - `shadcn/ui` (Phiên bản dựa trên **Base UI/MUI**, KHÔNG PHẢI Radix UI).
  - `zod`: Schema validation.

## 3. Architecture & Data Flow (Kiến trúc & Luồng dữ liệu)
- **Pattern:** Folder-by-feature (App Router), tách biệt logic ở `src/lib/`, UI components tái sử dụng ở `src/components/ui/`.
- **Primary Data Flow:** Client Component (Dashboard/Modal) ➔ Gọi Next.js Server Actions ➔ Bypass RLS (nếu cần bằng Service Role) hoặc dùng Auth Client ➔ Database Supabase (các schema `transactions`, `debts`, `assets`, `categories`).
- **Core Value:** Giao diện Dashboard Bento Grid hiện đại (Dark Mode "Premium"), thiết kế **VisionOS-inspired** (Premium Glassmorphism), tối ưu diện tích (Compact), chuyển User nhanh bằng thanh **Toggle một chạm**.
- **Animation Philosophy:** Hạn chế chuyển động thừa. Ưu tiên giao diện tĩnh (Static) để tăng độ sang trọng. Chỉ dùng hiệu ứng "thở" (breathing) cho các cảnh báo cực kỳ quan trọng (ví dụ: vượt ngưỡng ngân sách).

## 4. Key Logic (Các Lỗi & Kinh nghiệm sinh tử - Đọc kỹ)
- **[CRITICAL] Component Base UI:** Thư viện giao diện đang cài đặt là **Base UI**. Khi dùng các component bọc (như `DialogTrigger`), **TUYỆT ĐỐI KHÔNG DÙNG** thuộc tính `asChild`. 
- **[UPDATE] Debt & Asset Logic:** Tài sản (Assets) và Khoản nợ (Debts) không phân biệt owner cụ thể (luôn là Gia đình). Chỉ có Giao dịch (Transactions) mới phân chia `HIEU`/`LY`.
- **[NEW] Transaction Adjustment:** Mọi thao tác cập nhật số dư tiền mặt hoặc khởi tạo nợ (khi chọn Update Cash) đều tạo bản ghi `ADJUSTMENT` để minh bạch lịch sử dòng tiền.
- **[NEW] Accessibility & Console Errors:** Phải truyền trực tiếp `children` vào thuộc tính `render` của `DialogTrigger` (ép kiểu `as React.ReactElement`) để tránh lỗi lồng thẻ button gây đỏ console.
- **Compact UI Design:** Sử dụng bo góc `rounded-3xl` (24px) cho các thẻ lớn và `rounded-full` cho các nút điều hướng/header để tạo sự mềm mại.
- **[NEW] Deficit Warning UI:** Khi chi tiêu vượt thu nhập, thẻ so sánh không dùng animation pulsing mà dùng **Static Glow Shadow** (cam nhạt, lan tỏa 40px) và viền cam để cảnh báo.
- **Header Synchronization:** Các bộ chọn chức năng (Tháng/Năm, User Switcher) phải đồng bộ tuyệt đối về `padding`, `font-size (text-xs)`, `font-weight (semibold)` và `ring-1 border`.
- **Deploy Vercel:** Dự án bắt buộc cần file `vercel.json` định tuyến `outputDirectory: ".next"`.
- **[NEW] Timezone Vercel:** Các logic tính ngày tháng (như đầu tháng/cuối tháng) bắt buộc sử dụng chuẩn Timezone `Asia/Ho_Chi_Minh` (GMT+7) thay vì UTC local của server Vercel để tránh sai lệch dữ liệu vào đầu ngày.
- **[NEW] Zod Validation:** Tính năng import/export dữ liệu phải được Validate chặt chẽ bằng Zod schema v4 (sử dụng `.loose()`) trước khi nạp vào Supabase để bảo vệ Database.
- **[NEW] Admin Singleton:** Supabase admin client (`@supabase/supabase-js`) cần được thiết lập dạng Singleton Pattern để tránh memory leak.

## 6. Team Rules & Conventions (Quy tắc)
- **Naming:** `camelCase` cho hàm/biến, `PascalCase` cho Component, `snake_case` cho database columns.
- **Styling:** Ưu tiên **UI Premium** (Glassmorphism với `backdrop-blur-md`, gradient borders tinh tế `white/10` đến `white/20`, shadow `shadow-2xl`).
- **Navigation:** Mobile Bottom Navbar sử dụng nhãn tiếng Anh và thiết kế **Ultra-compact**.
- **Coding:** TypeScript strict mode, ES6+, Lambda, Shorthand logic. Phân tách rõ Client (`"use client"`) và Server components.
- **Code Format:** Chỉ cung cấp `diff`. Cấm Yapping.
- **Communication:** Ngôn ngữ Tiếng Việt, xưng hô Anh/Em. Phản hồi ngắn gọn.

## 8. Kế Hoạch Refactor Tối Ưu Hóa (Vercel + Supabase + Github)
*(Mục tiêu: Hoạt động nhanh, mượt, dễ bảo trì, dễ nâng cấp, giữ nguyên behavior)*

1. **Tối ưu Data Fetching & Caching (Vercel/Next.js Layer)**
   - Chuyển đổi từ gọi trực tiếp Server Actions chặn luồng sang sử dụng Next.js Cache (`unstable_cache` hoặc fetch options với `next: { tags: [...] }`).
   - Tận dụng `revalidateTag` thay vì `revalidatePath('/')` cục bộ nhằm chỉ clear đúng cache bị thay đổi, giúp server phản hồi tức thời.
   - Ứng dụng React Suspense boundaries thay vì chặn await toàn bộ Page để render UI layout trước (Streaming SSR).

2. **Kiến Trúc Database & API An Toàn (Supabase Layer)**
   - Di chuyển các complex mutations (như bán tài sản -> sinh giao dịch -> cập nhật tiền mặt) từ Server Actions Node.js xuống **PostgreSQL RPC (Stored Procedures)** hoặc **Triggers**.
   - Mục đích: Đảm bảo tính nhất quán dữ liệu (ACID/Transactions), loại bỏ việc sử dụng `SUPABASE_SERVICE_ROLE_KEY` (bypass RLS) ở tầng API Next.js.

3. **Cấu Trúc Code & Khả Năng Bảo Trì (Codebase Architecture)**
   - Tách biệt Data Access Layer (Các hàm gọi DB trực tiếp) và Service Layer (Chứa logic tính toán kinh doanh).
   - Đưa tất cả Zod Schema vào một folder dùng chung (`src/lib/schemas/`) để frontend (form validation) và backend (server actions) có thể đồng bộ.
   - Tự động hóa việc lấy types từ Supabase (Supabase CLI typegen) để tránh định nghĩa manual các Interface.

4. **Trải Nghiệm Người Dùng & UI Mượt Mà (Client Performance)**
   - Dùng `next/dynamic` để lazy-load các component nặng như `recharts`.
   - Áp dụng `useOptimistic` của React 19 tại các Server Action giúp giao diện cập nhật ngay lập tức trước khi server phản hồi, tạo cảm giác mượt mà tuyệt đối cho các thao tác thêm sửa xóa.
